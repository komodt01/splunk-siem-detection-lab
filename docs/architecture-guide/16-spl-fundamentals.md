# 16. SPL Fundamentals

## Purpose

This document introduces the foundational concepts and commands of the Splunk Search Processing Language (SPL). SPL enables analysts, administrators, detection engineers, and security architects to search, transform, summarize, and visualize indexed security events.

The objective is not to memorize commands. The objective is to understand how SPL converts security questions into repeatable analytical workflows that support investigations, reporting, monitoring, and detection engineering.

## Business Perspective

Organizations collect security telemetry so that analysts can answer meaningful questions about risk, suspicious activity, system behavior, and potential incidents.

Raw log retention alone does not provide sufficient business value. Security teams must be able to rapidly retrieve relevant events, isolate suspicious activity, identify patterns, measure trends, and communicate findings.

SPL provides the analytical layer that transforms stored telemetry into useful security information.

Effective SPL searches can help an organization:

* Reduce investigation time
* Detect suspicious activity
* Identify affected users and systems
* Support compliance reporting
* Measure security trends
* Validate security controls
* Improve incident response
* Create repeatable operational processes

Poorly designed searches can consume excessive resources, return misleading results, and increase analyst workload. Search design therefore has operational, architectural, and cost implications.

## What Is SPL?

The Search Processing Language is Splunk's language for retrieving and analyzing indexed data.

An SPL search typically begins by identifying a set of relevant events and then applies commands that filter, organize, transform, calculate, or visualize the results.

A basic search may look like:

```spl
index=security_lab
```

This search retrieves events stored in the `security_lab` index.

Additional criteria can narrow the result set:

```spl
index=security_lab sourcetype=linux_secure
```

This retrieves events stored in the `security_lab` index that were parsed using the `linux_secure` sourcetype.

A more specific search may include multiple metadata fields:

```spl
index=security_lab sourcetype=linux_secure host="KO" source="/var/log/auth.log"
```

This search retrieves Linux authentication events generated by the `KO` host and collected from `/var/log/auth.log`.

## Search Pipeline

SPL commonly uses the pipe character to send the results of one operation into the next command.

Example:

```spl
index=security_lab sourcetype=linux_secure
| table _time host source _raw
```

The base search retrieves the events.

The `table` command then organizes selected fields into columns.

The search pipeline can be understood as:

```text
Retrieve Events
       ↓
Filter Results
       ↓
Transform Data
       ↓
Summarize Activity
       ↓
Present Findings
```

Each command should serve a clear analytical purpose.

## Base Search

The base search identifies the initial event population.

A broad base search is:

```spl
index=security_lab
```

A more focused base search is:

```spl
index=security_lab sourcetype=linux_secure
```

A highly constrained base search is:

```spl
index=security_lab sourcetype=linux_secure host="KO" source="/var/log/auth.log"
```

Starting with indexed metadata is generally more efficient than retrieving a large event population and filtering it later in the pipeline.

## Time Range

Every Splunk search operates within a selected time range.

The time range may be selected through the Splunk interface or defined within SPL.

Examples include:

```spl
index=security_lab earliest=-24h
```

```spl
index=security_lab earliest=-7d
```

```spl
index=security_lab earliest=-30m
```

Time constraints improve search relevance and reduce the amount of data Splunk must process.

Analysts should select the narrowest reasonable time window that supports the investigative question.

## Search Terms

Search terms filter events based on values found in the raw event or extracted fields.

Example:

```spl
index=security_lab "Failed password"
```

This retrieves events containing the phrase `Failed password`.

Multiple terms can be combined:

```spl
index=security_lab "Failed password" sshd
```

Splunk returns events containing both terms.

Exact phrases should be enclosed in quotation marks.

## Boolean Operators

Boolean operators combine or exclude search conditions.

Common operators include:

* `AND`
* `OR`
* `NOT`

Example:

```spl
index=security_lab "Failed password" AND sshd
```

Example:

```spl
index=security_lab "Failed password" OR "Invalid user"
```

Example:

```spl
index=security_lab sshd NOT "Failed password"
```

Boolean logic should be grouped carefully to avoid unintentionally broad or misleading results.

## The Search Command

The `search` command filters events within the pipeline.

Example:

```spl
index=security_lab sourcetype=linux_secure
| search "Failed password"
```

The same condition can often be placed in the base search:

```spl
index=security_lab sourcetype=linux_secure "Failed password"
```

Placing restrictive terms in the base search is usually preferable because it can reduce the number of events entering the search pipeline.

The explicit `search` command remains useful when filtering results after another command or transformation.

## The Table Command

The `table` command displays selected fields as columns.

Example:

```spl
index=security_lab sourcetype=linux_secure
| table _time host source sourcetype _raw
```

This improves readability by showing only fields relevant to the investigation.

The `table` command is useful for:

* Reviewing individual events
* Presenting investigation evidence
* Validating field extraction
* Creating readable reports
* Removing unnecessary information from results

The command is primarily a presentation tool and is often best used near the end of a search.

## The Fields Command

The `fields` command includes or excludes fields from search results.

Example:

```spl
index=security_lab sourcetype=linux_secure
| fields _time host source _raw
```

Fields can also be excluded:

```spl
index=security_lab sourcetype=linux_secure
| fields - source sourcetype
```

Reducing unnecessary fields may improve readability and, in some situations, search efficiency.

## The Stats Command

The `stats` command calculates aggregate values across a set of events.

Example:

```spl
index=security_lab sourcetype=linux_secure
| stats count
```

This returns the total number of matching events.

Events can be grouped by a field:

```spl
index=security_lab sourcetype=linux_secure
| stats count by host
```

This returns the number of events associated with each host.

Another example is:

```spl
index=security_lab sourcetype=linux_secure
| stats count by source
```

The `stats` command supports analytical questions such as:

* How many events occurred?
* Which hosts generated the most activity?
* Which users experienced the most authentication failures?
* Which source IP addresses appear most frequently?
* What is the average, minimum, or maximum value of a field?

## The Sort Command

The `sort` command organizes results in ascending or descending order.

Example:

```spl
index=security_lab sourcetype=linux_secure
| stats count by host
| sort - count
```

The minus sign indicates descending order.

Ascending order can be requested with:

```spl
| sort count
```

Sorting helps analysts quickly identify the most frequent or highest-risk values.

## The Rename Command

The `rename` command changes field names for readability.

Example:

```spl
index=security_lab sourcetype=linux_secure
| stats count by host
| rename host AS "System", count AS "Event Count"
```

Renaming fields can improve dashboards, reports, and executive-facing outputs.

Field names used for detection logic should remain clear and consistent. Presentation-oriented naming should generally occur near the end of the search.

## The Where Command

The `where` command filters results using evaluated expressions.

Example:

```spl
index=security_lab sourcetype=linux_secure
| stats count by host
| where count > 10
```

This returns only hosts with more than ten matching events.

The `where` command is useful when filtering calculated values or applying comparisons involving multiple fields.

Examples of supported comparisons include:

* Greater than
* Less than
* Equal to
* Not equal to
* Logical expressions
* Mathematical expressions

Whenever possible, simple event filters should be placed in the base search. The `where` command is most useful after fields have been calculated or transformed.

## The Eval Command

The `eval` command creates or modifies fields.

Example:

```spl
index=security_lab sourcetype=linux_secure
| eval environment="Lab"
| table _time host environment _raw
```

The `eval` command can support:

* Conditional logic
* Mathematical calculations
* String manipulation
* Field normalization
* Risk scoring
* Categorization
* Time calculations

Example:

```spl
index=security_lab sourcetype=linux_secure
| eval event_category=if(searchmatch("Failed password"), "Authentication Failure", "Other")
| table _time host event_category _raw
```

Calculated fields should be clearly named and documented when they are used in operational detections.

## The Timechart Command

The `timechart` command displays aggregated activity over time.

Example:

```spl
index=security_lab sourcetype=linux_secure
| timechart count
```

This shows the number of matching events across time intervals.

A more focused example is:

```spl
index=security_lab sourcetype=linux_secure "Failed password"
| timechart count
```

Time-based analysis helps analysts identify:

* Spikes in authentication activity
* Repeated failures
* Unusual usage periods
* Changes in system behavior
* Trends that may not be visible in individual events

The time interval can be controlled with the `span` argument:

```spl
index=security_lab sourcetype=linux_secure
| timechart span=1h count
```

The appropriate span depends on the search time range and the investigative question.

## The Dedup Command

The `dedup` command removes duplicate results based on one or more fields.

Example:

```spl
index=security_lab sourcetype=linux_secure
| dedup host
| table host
```

This returns one result for each unique host.

The command can help analysts identify unique users, systems, addresses, or other entities.

It should be used carefully because removing repeated events may hide frequency or sequence information that is important during an investigation.

## The Top Command

The `top` command identifies the most common values for a field.

Example:

```spl
index=security_lab sourcetype=linux_secure
| top host
```

The output includes count and percentage values.

This command can quickly answer questions such as:

* Which hosts generate the most events?
* Which users appear most frequently?
* Which source addresses are most active?
* Which event categories dominate the data?

## The Rare Command

The `rare` command identifies the least common values for a field.

Example:

```spl
index=security_lab sourcetype=linux_secure
| rare host
```

Rare activity is not automatically malicious, but it may warrant further investigation.

Security analysis frequently requires examining both frequent behavior and unusual behavior.

## Initial Authentication Searches

The current lab uses `/var/log/auth.log` with the `linux_secure` sourcetype.

The following searches provide an initial understanding of the available data.

Retrieve all Linux authentication events:

```spl
index=security_lab sourcetype=linux_secure
```

View core event metadata:

```spl
index=security_lab sourcetype=linux_secure
| table _time host source sourcetype _raw
```

Count all authentication events:

```spl
index=security_lab sourcetype=linux_secure
| stats count
```

Count events by host:

```spl
index=security_lab sourcetype=linux_secure
| stats count by host
| sort - count
```

Find failed-password events:

```spl
index=security_lab sourcetype=linux_secure "Failed password"
```

Find successful SSH authentications:

```spl
index=security_lab sourcetype=linux_secure "Accepted password"
```

Find invalid-user activity:

```spl
index=security_lab sourcetype=linux_secure "Invalid user"
```

Find sudo-related activity:

```spl
index=security_lab sourcetype=linux_secure sudo
```

Display authentication volume over time:

```spl
index=security_lab sourcetype=linux_secure
| timechart count
```

These searches should be treated as investigative starting points rather than complete detections.

## Search Development Process

A reliable SPL development process should follow a structured sequence.

### Define the Question

Begin with a clear security or operational question.

Example:

```text
How many failed SSH authentication attempts occurred during the last twenty-four hours?
```

### Identify the Data Source

Determine which index, sourcetype, source, or host contains the required telemetry.

### Validate Raw Events

Review representative events before relying on extracted fields.

### Narrow the Search

Use metadata, time range, search terms, and fields to reduce irrelevant results.

### Aggregate or Transform

Use commands such as `stats`, `eval`, or `timechart` to identify patterns.

### Validate the Results

Confirm that the search returns the expected events and does not exclude relevant activity.

### Present the Findings

Use `table`, field renaming, reports, or visualizations to communicate results clearly.

### Document the Logic

Record the purpose, assumptions, data dependencies, limitations, and tuning considerations.

## Search Performance Considerations

Search performance affects analyst productivity, infrastructure capacity, and operational cost.

Architects and search developers should:

* Use the narrowest appropriate time range
* Specify indexes explicitly
* Filter by metadata early
* Avoid unnecessarily broad wildcard searches
* Limit returned fields where practical
* Aggregate only the events required
* Avoid expensive commands when simpler logic is available
* Test searches using realistic data volumes
* Review scheduled-search frequency
* Prevent redundant reports and alerts

A search that performs acceptably in a small lab may become expensive when applied to enterprise-scale data.

## Security Considerations

Access to SPL and indexed data should follow least-privilege principles.

Organizations should consider:

* Restricting access to sensitive indexes
* Protecting authentication and identity data
* Preventing unauthorized access to raw events
* Controlling who can create scheduled searches
* Reviewing searches that expose personal or regulated information
* Logging administrative and search activity
* Separating analyst, administrator, and executive access
* Protecting saved searches and detection logic from unauthorized changes

Search results may contain usernames, IP addresses, hostnames, system details, and other sensitive information.

## Operational Considerations

Operationally useful SPL should be understandable, maintainable, and repeatable.

Searches should include:

* A clearly defined purpose
* Consistent field names
* Appropriate time ranges
* Documented data dependencies
* Expected results
* Known limitations
* An owner
* A review schedule
* Version control where practical

Complex searches without documentation create operational risk because analysts may not understand their assumptions or intended use.

## Architecture Considerations

SPL is influenced by upstream architecture.

Search effectiveness depends on:

* Correct timestamp extraction
* Reliable data ingestion
* Appropriate index design
* Consistent sourcetypes
* Accurate field extraction
* Standardized field names
* Data retention
* Access controls
* Search-head capacity
* Indexer performance
* Data normalization

Search logic cannot fully compensate for missing, incomplete, or poorly structured telemetry.

## Tradeoffs

SPL design involves several tradeoffs.

Broad searches provide greater visibility but require more compute resources.

Highly restrictive searches improve performance but may exclude relevant events.

Complex transformations may improve analytical value but reduce maintainability.

Frequent scheduled searches improve detection speed but increase infrastructure utilization.

Long time ranges support historical investigation but increase search cost.

Architects must balance fidelity, speed, complexity, maintainability, and cost.

## Best Practices

* Begin with a specific investigative question.
* Always specify the relevant index.
* Use metadata filters early.
* Select an appropriate time range.
* Validate representative raw events.
* Use readable and consistently formatted SPL.
* Add one transformation at a time.
* Validate results after each command.
* Keep presentation commands near the end.
* Document assumptions and dependencies.
* Test searches against known positive and negative examples.
* Review scheduled searches for performance and continued relevance.
* Avoid treating a simple keyword search as a production-ready detection.

## Interview Questions

### What is SPL?

SPL is Splunk's search and analytics language for retrieving, filtering, transforming, aggregating, and visualizing indexed event data.

### Why should an SPL search specify an index?

Specifying an index reduces the search scope, improves performance, and ensures that the search targets the intended data population.

### What is the purpose of the pipe character?

The pipe passes the results of one search operation into the next SPL command.

### What is the difference between table and stats?

`table` presents individual event fields in columns. `stats` aggregates events and calculates values such as counts, averages, minimums, and maximums.

### Why should filtering occur early?

Early filtering reduces the number of events processed by later commands, improving performance and limiting irrelevant results.

### What makes an SPL search production ready?

A production-ready search has a defined purpose, validated data sources, tested logic, acceptable performance, documented assumptions, clear ownership, and known tuning considerations.

### Can a successful SPL search still produce a poor detection?

Yes. The syntax may be valid while the logic is incomplete, noisy, dependent on poor-quality data, or unable to distinguish legitimate from malicious activity.

## Lessons Learned

A successful search is not necessarily a useful search.

SPL commands should be selected based on the question being answered rather than used simply because they are available.

Metadata such as index, host, source, and sourcetype provides the most reliable starting point for narrowing search scope.

Validation of raw events remains necessary even when extracted fields are available.

Search performance must be considered during initial design rather than only after deployment.

SPL becomes more useful when it is treated as an analytical process instead of a list of commands to memorize.

## Architect's Perspective

Security architects do not need to write every operational search, but they should understand how search design depends on data architecture, governance, scalability, and detection objectives.

An architect should be able to evaluate whether the required telemetry exists, whether it is searchable, whether relevant fields are available, whether the search can scale, and whether the results support an actual business or security outcome.

SPL is where ingestion architecture becomes operational value.

A well-designed search demonstrates that the organization has collected the right data, structured it correctly, retained it appropriately, and made it available to authorized users who can turn it into actionable intelligence.

## Key Takeaways

* SPL transforms indexed events into actionable security information.
* Searches should begin with a clear investigative or business question.
* Index, time range, host, source, and sourcetype should be used to narrow results.
* Pipes create a sequence of analytical operations.
* Commands such as `table`, `stats`, `sort`, `where`, `eval`, and `timechart` support different stages of analysis.
* Search performance is an architectural and operational concern.
* Keyword searches are useful for exploration but are not automatically production-ready detections.
* Effective SPL depends on reliable ingestion, accurate timestamps, correct parsing, and meaningful fields.
* SPL should be validated, documented, governed, and maintained like other operational security content.
