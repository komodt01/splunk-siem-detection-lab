# Splunk Search Processing Language (SPL)

## Purpose

Splunk Processing Language (SPL) is the primary language used to search, filter, transform, analyze, and visualize indexed data. It enables security analysts, engineers, and architects to investigate events, identify threats, validate detections, and build operational dashboards.

This document introduces SPL concepts that will be expanded throughout this project.

---

# What Is SPL?

SPL is Splunk's search language.

Unlike SQL, which queries relational databases, SPL is optimized for searching and analyzing machine-generated event data.

It allows users to:

- Search events
- Filter results
- Aggregate data
- Calculate statistics
- Create reports
- Build dashboards
- Support investigations
- Develop detections

---

# Basic Search Structure

A typical SPL search consists of:

- Index selection
- Search criteria
- Filters
- Transforming commands
- Reporting commands

Example:

```spl
index=main
```

A more targeted example:

```spl
index=windows EventCode=4625
```

---

# Common Search Components

### Index

Limits the search to one or more indexes.

Example:

```spl
index=linux
```

---

### Host

Searches events generated by a specific system.

Example:

```spl
host=server01
```

---

### Source

Searches a specific log file or input.

Example:

```spl
source="/var/log/auth.log"
```

---

### Source Type

Searches events based on their format.

Example:

```spl
sourcetype=WinEventLog:Security
```

---

### Time Range

Limits searches to a defined period.

Examples include:

- Last 15 minutes
- Last hour
- Last 24 hours
- Last 7 days

Restricting the time range improves search performance.

---

# Search Pipeline

SPL commands execute from left to right.

Example:

```spl
index=windows EventCode=4625
| stats count by Account_Name
```

The first stage retrieves events.

The second stage summarizes the results.

Understanding the pipeline is essential for writing efficient searches.

# Security Use Cases

SPL supports many security activities, including:

---

# Performance Considerations

Search performance improves when:

- Searching only required indexes
- Restricting time ranges
- Filtering early
- Avoiding unnecessary wildcard searches
- Returning only required fields

Efficient searches reduce infrastructure load and improve analyst productivity.

---

# Security Considerations

Architects should consider:

- Role-based search permissions
- Sensitive data exposure
- Search auditing
- Search performance
- Access to restricted indexes

Poorly designed searches can expose sensitive information or unnecessarily consume system resources.

---

# Architecture Considerations

SPL is more than a search language.

It represents the interface between stored security telemetry and operational decision making.

Architectural decisions affecting SPL include:

- Index design
- Source type consistency
- Metadata quality
- Retention strategy
- Search performance
- Detection design

Well-designed data ingestion simplifies SPL development.

---

# Tradeoffs

### Broad Searches

Benefits

- Maximum visibility
- Useful during investigations

Tradeoffs

- Slower performance
- Increased resource consumption
- More irrelevant results

---

### Targeted Searches

Benefits

- Faster execution
- Better efficiency
- More precise investigations

Tradeoffs

- Possible missed context
- Requires better understanding of available data

---

# Best Practices

- Filter early.
- Search only required indexes.
- Limit time ranges.
- Return only needed fields.
- Use descriptive saved searches.
- Validate search results.
