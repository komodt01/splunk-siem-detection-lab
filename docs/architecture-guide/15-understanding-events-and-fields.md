# 15. Understanding Events and Fields

## Purpose

This document explains how Security Information and Event Management (SIEM) platforms transform raw log data into structured, searchable security events. Understanding events and fields is fundamental to effective security operations because every search, dashboard, alert, report, and detection begins with event data.

Rather than viewing logs as plain text, security professionals should understand them as structured records containing valuable security information that supports investigation, detection, compliance, and threat hunting.

## Business Perspective

Modern organizations generate enormous volumes of security telemetry from servers, cloud platforms, identity providers, firewalls, applications, endpoint security tools, and network devices.

Individually, these logs provide limited value.

The business derives value only when those logs become searchable, correlated, and understandable.

Security teams rely on structured events to answer questions such as:

- What happened?
- When did it happen?
- Who performed the activity?
- Which systems were involved?
- Was the activity expected or suspicious?
- Does the activity indicate malicious behavior?

Without structured event data, organizations lose visibility into their security environment and significantly increase the time required to investigate incidents.

## What Is a Security Event?

A security event is a single recorded activity generated by a system, application, cloud service, network device, or security control.

Each event represents one action occurring at a specific point in time.

Examples include:

- User authentication
- Failed authentication
- SSH connection
- Privilege escalation
- File access
- Firewall allow
- Firewall deny
- Process execution
- Password change
- VPN connection
- Malware detection

Every event contributes a small piece of information. During an investigation, analysts combine thousands or even millions of individual events to reconstruct what occurred during an attack.

## Raw Data Versus Structured Data

Operating systems and applications typically generate logs as plain text.

For example:

```
Jul 14 08:20:31 KO sshd[2298]: Failed password for invalid user admin from 10.1.1.50 port 50500 ssh2
```

Although a human can interpret this message, computers require structure to efficiently search and analyze the data.

Splunk transforms this raw log entry into searchable metadata and fields such as:

- _time
- host
- source
- sourcetype
- index
- username
- source IP
- action
- process

Rather than searching text, analysts search structured information.

This transformation is one of the primary reasons SIEM platforms provide significant operational value.

## Event Anatomy

Every event consists of multiple components that provide context during an investigation.

### Timestamp

Identifies when the event occurred.

Accurate timestamps are essential for building timelines and correlating activity across systems.

### Host

Identifies the system that generated the event.

Examples include:

- Linux servers
- Windows servers
- Domain controllers
- Firewalls
- Cloud instances

Host information allows analysts to determine where an event originated.

### Source

Identifies the original log file, device, or input responsible for generating the event.

Examples include:

- /var/log/auth.log
- Windows Security Event Log
- AWS CloudTrail
- Azure Activity Logs

The source helps investigators determine the origin of collected telemetry.

### Sourcetype

Defines how Splunk should interpret and parse incoming data.

Examples include:

- linux_secure
- syslog
- aws:cloudtrail
- wineventlog

Selecting the appropriate sourcetype ensures accurate field extraction and improves search quality.

### Index

Identifies the logical storage location where events are indexed.

Indexes separate different categories of data while supporting retention policies, access controls, and search performance.

### Raw Event

The raw event contains the original log entry exactly as it was received.

Preserving raw data supports forensic investigations and enables organizations to validate parsing and field extraction.

### Extracted Fields

Splunk extracts structured values from raw events to make searches more efficient.

Fields allow analysts to search specific attributes rather than scanning unstructured text.

### Metadata

Metadata describes information about the event rather than the activity itself.

Examples include:

- Host
- Source
- Sourcetype
- Index

Although often overlooked, metadata plays an important role in organizing and searching security data.

## Why Metadata Matters

Metadata improves both operational efficiency and system scalability.

Architects use metadata to support:

- Efficient searches
- Logical data organization
- Data retention strategies
- Role-based access controls
- Performance optimization
- Compliance reporting

Well-designed metadata reduces search complexity while improving investigative efficiency.

## What Are Fields?

Fields describe specific attributes contained within an event.

Examples include:

- username
- source IP
- destination IP
- hostname
- action
- process
- protocol
- status
- bytes transferred
- application

Fields allow analysts to ask meaningful investigative questions without manually reading every log entry.

## Default Splunk Fields

Splunk automatically provides several metadata fields for nearly every indexed event.

These include:

- _time
- host
- source
- sourcetype
- index
- _raw

These fields form the foundation of almost every Splunk search and provide the context necessary for filtering, reporting, and investigation.

## Extracted Fields

Beyond default metadata, Splunk extracts additional fields based on the event format and associated sourcetype.

Field extraction may occur automatically through built-in knowledge objects or through custom configurations created by administrators.

Organizations may also define additional field extractions to support proprietary applications and specialized log formats.

Accurate field extraction improves reporting, dashboards, detections, and investigative capabilities.

## Why Fields Matter

Fields transform unstructured text into actionable security intelligence.

For example, an analyst may ask:

- Show all failed SSH logins.
- Show authentication failures by user.
- Show activity from a specific IP address.
- Show events generated by a particular host.
- Show login activity during the last twenty-four hours.

Without structured fields, these searches become significantly more difficult and less reliable.

## Data Quality

The quality of security analytics depends directly on the quality of collected data.

Examples include:

Poor timestamps result in inaccurate investigation timelines.

Incorrect host names make asset identification more difficult.

Missing usernames prevent analysts from identifying affected accounts.

Improper parsing reduces search accuracy.

Incomplete field extraction decreases detection quality.

Poor data quality cannot be corrected simply by collecting more logs.

Security architects should prioritize high-quality telemetry over high-volume telemetry.

## Architectural Considerations

Architectural decisions made during data ingestion directly affect downstream security operations.

Architects should evaluate:

- Time synchronization
- Parsing accuracy
- Field extraction quality
- Naming standards
- Data consistency
- Retention requirements
- Storage efficiency
- Search performance
- Scalability

Every decision influences the effectiveness of detections, investigations, dashboards, and compliance reporting.

## Security Architect Perspective

Security architects should view events as structured information rather than log files.

The objective is not to collect the greatest possible quantity of logs, but to collect high-quality telemetry that enables rapid detection, effective investigations, accurate reporting, and informed business decisions.

Every logging decision should improve visibility, increase detection fidelity, reduce investigation time, and support organizational security objectives.

An effective SIEM architecture is built upon reliable, well-structured event data.

## Key Takeaways

- Events are the fundamental unit of security monitoring.
- Every event represents a single recorded activity.
- Structured fields transform raw logs into searchable security information.
- Metadata supports organization, scalability, and search efficiency.
- High-quality data produces higher-quality detections and investigations.
- Effective SIEM implementations prioritize data quality over data quantity.
- Every search, report, dashboard, alert, and detection begins with well-structured event data.
